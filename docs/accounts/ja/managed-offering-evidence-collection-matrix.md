# Managed Offering エビデンス収集マトリクス

> このページでわかること: `takosumi.managed-offering-readiness@v1` の各 evidence type を、operator がどの source から集め、どの private artifact に保存し、どの structured field / pass criteria / redaction rule で bundle に入れるか。

この matrix は collection guide です。operator が実行する正本 workflow は `takos-private` の `managed-offering:workspace` / `managed-offering:validate` / `managed-offering:topology:preflight` / `managed-offering:topology:merge` / `managed-offering:public-summary` / `managed-offering:public-summary:validate` / `managed-offering:audit` です。下層の validation contract は [`managed-offering-launch-readiness.md`](./managed-offering-launch-readiness.md) と `takosumi launch-readiness validate --file <json>` です。raw tenant data、secret、 provider account ID、Stripe object ID、legal draft、support mailbox content は public summary に出さず、private evidence store の `ref` だけに紐づけます。

## Collection Rules

| Rule                             | Requirement                                                                                                                       |
| -------------------------------- | --------------------------------------------------------------------------------------------------------------------------------- |
| Source system                    | Accounts route / dashboard / Stripe / telemetry / CI / vault / legal system など、operator が実際に確認した system を記録する     |
| Collection command / API / probe | command transcript、HTTP route、dashboard export、synthetic probe、sign-off record のいずれかを private artifact に保存する       |
| Private artifact/ref             | `vault://managed-readiness/<env>/<run>/<type>.json`、`artifact://...`、`dashboard://...`、`approval://...` など stable ref を使う |
| Required structured fields       | `launch-readiness template` が出す field を埋める。placeholder、`example.*`、短すぎる summary は不可                              |
| Pass criteria                    | live launch 判断に使える record だけ `status: "passed"` にする。dry-run のみの record は明示的に許可された行以外は不可            |
| Public redaction                 | public summary は「何をどの環境で検証したか」だけにし、raw ID / PII / secret / provider detail は出さない                         |

Cloudflare 上の Takosumi Accounts は Worker-only substrate として扱います。 Workers.dev の `/healthz` が `persistence:"d1+r2"` を返すことは bootstrap evidence ですが、launch topology / TLS evidence ではありません。staging / production topology の DNS/TLS / health probe evidence には、 `takosumi` の `deploy:accounts-cloudflare:ensure-dns -- --check --fail-on-not-ready` で live `accounts.takosumi.com` proxied CNAME を確認した JSON と、 `deploy:accounts-cloudflare:probe -- --fail-on-not-ready` で custom-domain health/OIDC を確認した `takosumi.cloudflare-accounts-probe@v1` JSON を private artifact として保存し、Accounts `wranglerConfigRef` / `healthProbeEvidenceRef` / TLS evidence ref に紐づけます。

## Launch Opening Audit

After the private bundle and public summary are generated, the `takos-private` audit wrapper must be the last local command before opening public managed access:

```bash
cd takos-private
bun run managed-offering:audit -- \
  --environment <staging|production> \
  --date <YYYY-MM-DD> \
  --evidence-ref <private evidence ref> \
  --approval-ref <separate approval ref>
```

The wrapper delegates to the root `audit:managed-offering-live-readiness` implementation. It validates the private bundle, validates the public summary artifact, dry-runs `takosumi accounts serve --managed-offering-access open` with the same canonical digest, private evidence ref, separate approval ref, and sanitized public summary, and emits `accountsServeManagedOfferingArgs` for the real Accounts service command.
