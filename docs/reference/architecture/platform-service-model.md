# Platform Service Model {#platform-service-model}

Operator platform service compatibility model:
[Platform Services](../platform-services.md).

PlatformService は Space-visible な service capability です。operator distribution
または product distribution が inventory entry を offer し、install / deploy 時に
BindingSelection を解決します。Takosumi は resolved binding snapshot を
Deployment record に保存します。

## Platform Service Path Grammar {#platform-service-path-grammar}

```text
segment = [a-z][a-z0-9-]{0,62}
path    = segment "." segment "." segment ("." segment)*
```

Rules:

- min segments: 3
- max segments: 8
- max path length: 255
- empty segments are invalid
- service roots are defined by the operator distribution or product distribution
  that offers the PlatformService.

Takosumi defines its account-layer service roots in its distribution
specification; start from [Takosumi](../accounts.md).

## Inventory And Materialization {#inventory-and-materialization}

### PlatformService {#platform-service}

Inventory declares what can be consumed.

```yaml
PlatformService:
  id: service_database_primary
  path: database.primary.connection
  spaceId: space_acme_prod
  owner:
    kind: operator
    id: reference-operator
  capability: postgres.connection
  sensitivity: secret-bearing
  accessModes:
    - read
    - read-write
  freshness:
    state: fresh | stale | revoked | unknown
    observedAt: ...
```

### Materialization {#materialization}

Runtime data is generated by operator-owned materialization.

```yaml
BindingMaterialization:
  platformServiceId: service_database_primary
  secretRefs: [secret://operator/databases/acme-prod/api]
  endpointRefs: []
  authorizationRefs: []
  runtimeHandles: []
```

Takosumi records the selected binding snapshot; external resource state and
credential rotation stay with the operator distribution.

## Space-Scoped Resolution {#space-scoped-resolution}

platform service resolution is always Space-scoped:

```text
1. Space-visible PlatformService inventory
2. operator policy and account-plane grants
3. selected binding snapshot, or required-binding failure
```

deployment-local object scope, generated scope, group / environment scope, and
cross-space service sharing are future RFC vocabulary.

## Active Path Uniqueness {#active-path-uniqueness}

An operator may enforce one active provider for a given `(spaceId, path)` to keep
exact service selection unambiguous. This is inventory policy, not a source-file
grammar rule.

## Freshness {#freshness}

```text
fresh:
  usable

stale:
  policy decides allow-with-warning, require-refresh, or deny

revoked:
  deny

unknown:
  policy decides require-refresh or deny
```
