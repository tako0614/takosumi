# Takosumi Core Conformance

> [`core-spec.md`](./core-spec.md) (2026-06-08 改訂: Space 直下の OpenTofu Capsule DAG + Provider Template / Provider Env Set 正本) に対する
> 適合状況 (conformance / extension candidates) を記録します。本書は正本 spec ではなく、実装と docs の現状 map です。
> model、schema、API、UI、billing、backup、implementation phases の正本は `core-spec.md` にあります。

## 状態区分

- **conformant**: 現状の code/test で満たしている。
- **implementation gap**: 正本 model に入ったが、現 code/test では未実装または部分実装。
- **extension candidate**: 正本 model の上に追加できる将来拡張。現在の適合性を下げる不足ではない。

## Production Evidence Boundary

このページの `conformant` は repo 内の code/test/docs が core spec に適合しているという意味であり、公開 hosted
managed offering の開放証跡ではありません。公開 managed offering は operator が live 環境で採取した別証跡を必要とします。

- production hardening evidence:
  `cd takosumi && bun run production-hardening:evidence -- <manifest.json>` と
  `cd takosumi && bun run production-hardening:gates -- <manifest.json> --url <gate-url> --require-enforced`
  に加え、public-safe row 用の
  `cd takosumi && bun run production-hardening:gates -- <manifest.json> --url <gate-url> --require-enforced --markdown-row`
  出力
- managed offering launch readiness:
  `takosumi run launch-readiness validate --file <private-readiness.json>`、
  `takosumi run launch-readiness public-summary --file <private-readiness.json> --evidence-ref <private-ref> --public-summary <reviewed-text> --markdown-row`、
  `takosumi run launch-readiness public-summary validate --file <summary.json> --readiness-file <private-readiness.json>`、および
  同じ platform worker の accounts-plane gate を開く
  `takosumi run accounts serve --managed-offering-access open`

operator evidence が未投入の場合でも local conformance は満たし得ますが、その状態は public access ready ではありません。

## Implementation Notes

- Drizzle physical tables use compact JSON ledger columns; `core-spec.md` names the logical records, and docs tests now
  cover every deploy-control D1 table plus logical row columns, key constraints, and logical lookup indexes.
- `artifacts` is implemented as a pointer ledger table in logical / D1 / Postgres Drizzle schemas, Worker D1 bootstrap,
  and store contract paths. Artifact bytes remain in R2 buckets; D1 stores only run-scoped object key / digest / size
  metadata.
- Source-scoped `source_sync` rows use `sourceId` and no longer overload `installationId`; old rows may still be read via
  the backward-compatible store fallback until deployed ledgers have run the additive `source_id` migration.
- Current physical source tree intentionally differs from the idealized module sketch: service domains live under
  `src/service/*`, Worker entrypoints under `worker/src/*`, the dashboard under `dashboard/`, runner code under
  `runner/`, and shared domain packages under `packages/*`. `core-spec.md` names this as the canonical
  physical layout.
- `core-spec.md` uses `core` / `files` / `talk` as an illustrative Space DAG and install-flow example. The implemented
  first-party OpenTofu Capsule catalog in this repo is currently `core`, `cloudflare-r2-storage`,
  `cloudflare-worker-service`, `cloudflare-static-site`, and `aws-s3-storage`; `talk` and `files` are example
  Git-installed service names, not conformant bundled module ids.
- Provider Template / Provider Env Set tables in `core-spec.md` are mirrored by Drizzle, Worker D1 bootstrap, and
  D1/Postgres/in-memory stores. Retired provider-pack/pin storage code is migration compatibility only, not public v1
  surface.

## Future Extensions

1. Extend provider-specific temporary vending to additional providers as they are added to the root-only arg mapping.
2. Official D1 schema-migration provider (spec 7.3 の昇格典型対象、Capsule 内 DB schema migration を apply 中に実行する
   resource を提供し Cloudflare API のみを呼ぶ) — planned / 未実装。
3. `Run.type="restore"` (Backup からの state 復元) — **schema reserved / 未実装 (planned)**。現状は `RunType` union /
   OpenAPI `Run.type` enum / store の `putRestoreRun` / `getRestoreRun` accessor だけが存在し、producer・queue action
   (`OpenTofuRunAction` に restore は含まない)・controller handler・API route は無いため、restore Run は作成・dispatch
   されない。restore は Backup から tfstate を上書きする破壊的操作なので、backup と対称に出荷する前に approval gate /
   state-generation rollback / downstream stale propagation / runner `restore` action / policy 評価を設計する必要がある。
   それまで run queue consumer は未対応 action に対し fail-closed (`run_queue_consumer_test.ts` が restore action message を
   invalid として ack/drop することを検証する)。
