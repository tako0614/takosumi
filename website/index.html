Manifest-driven
self-hostable PaaS.

JSON-LD Manifest 1 つで、 Cloudflare Workers / Kubernetes / bare-metal / 自前 runtime まで multi-substrate に deploy。 kernel は apply pipeline だけに専念し、 workflow / identity / billing / project convention を持たない pure layer。

5 分 quickstart → ドキュメントを読む
$ deno install -gA -n takosumi jsr:@takos/takosumi-cli

Manifest だけ書く

1 つの YAML / JSON で resources[] を宣言。 shape + provider + spec の三組で記述するだけ。 残りは kernel が DAG 解決して apply。

Substrate-neutral

kernel と runtime-agent は RuntimeAdapter で Deno / Node / Workers / Bun を吸収。 同じ bytes が Cloudflare にも Kubernetes にも、 自前 VM にも乗る。

Plugin-based

Shape catalog は operator が選ぶ。 公開 DNS provider を抜く、 AWS を足す、 自前 connector を JSR から install。 vendor lock-in は 構造的に発生しない。

Quickstart

## manifest.yml
apiVersion: "1.0"
kind: Manifest
metadata:
  name: hello-selfhost
resources:
  - name: hello
    shape: "web-service@v1"
    provider: "@takos/selfhost-docker-compose"
    spec:
      image: nginx:alpine
      port: 80
      scale: { min: 1, max: 1 }

$ takosumi deploy ./manifest.yml
  → applied web-service@v1#hello — http://localhost:18080

"持たないもの" list

  • workflow runner — takosumi-git 等 helper product 側
  • identity / billing — Takosumi Accounts 側
  • project convention — operator distribution 側
  • vendor-lock の publisher signing — operator-pinned digest のみ

これらを 持たない ことが、 kernel が任意の substrate に乗る 唯一の理由。 詳細は architecture overview。

始める

JSR で 1 行 install、 manifest を書いて takosumi deploy。
